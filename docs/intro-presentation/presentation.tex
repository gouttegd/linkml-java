\section{Why do we need a LinkML runtime library for Java?}

\begin{frame}[fragile]
  \frametitle{Isn't generated code enough?}

  LinkML(-Py) can already generate Java code from a LinkML schema:

  \begin{columns}
    \begin{column}{.45\textwidth}
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
classes:
  Point:
    attributes:
      name:
      x:
        range: integer
      y:
        range: integer
\end{Verbatim}
    \end{column}
    \begin{column}{.45\textwidth}
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
public class Point {
    String name;
    int x;
    int y;
}
\end{Verbatim}
    \end{column}
  \end{columns}

  \medskip
  Then we can read or write instance data using any available JSON or
  YAML library for Java, such as Jackson:

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
ObjectMapper mapper = new ObjectMapper(new YAMLFactory());
Point p = mapper.readValue(new File("data.yaml"), Point.class);
// ...
mapper.writeValue(new File("data.yaml"), p);
\end{Verbatim}

  \medskip
  Isn't that all we need?

  \visible<2>{
  \medskip
  -- Well, no. This would only work for simple schemas that don't use
  any LinkML-specific features.
  }

  \note<1>{First, why do we even need a runtime library for LinkML?

  Using LinkML-Py, we can already generate Java classes that represent a
  LinkML model. With that, we should be able to use a standard JSON or
  YAML library to easily deserialise a JSON or YAML file into an
  instance of the Java class that represents a LinkML class, and then
  manipulate that object as any other Java object.

  So we should already have everything we need, right?

  No. Using a standard, non-LinkML-aware library would work, but only
  with the most simple schemas --~not simple in the sense that they
  don't contain a lot of classes, but in the sense that they don't use
  any LinkML-specific features.

  I'll show you two real-world examples.}
\end{frame}

\begin{frame}[fragile]
  \frametitle{Limitations of generated code: the case of KGCL}

  \begin{columns}[t]
    \begin{column}{.45\textwidth}
      KGCL ``change'' classes:
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
classes:
  Change:
    slots:
    - id
    - type
  SimpleChange:
    is_a: Change
    slots:
    - old_value
    - new_value
  # ...
\end{Verbatim}
    (More than 50 subclasses.)
    \end{column}
    \begin{column}{.45\textwidth}
Sample KGCL file:
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
- id: CHANGE:001
  type: NodeRename
  old_value: foo
  new_value: bar
  about_node: GO:0005635
- id: CHANGE:002
  type: NodeObsoletion
  about_node: GO:0005634
\end{Verbatim}
    (A list of \emph{Change} objects of various, not-known-in-advance
    types.)
    \end{column}
  \end{columns}

  \bigskip
  We want to be able to do something like:

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
List<Change> changes = mapper.readValues(new File("data.yaml"), Change.class);
\end{Verbatim}

  But this requires explicit support for \emph{LinkML type designator
  slots}, which the Jackson library obviously cannot have.

  \note{The first example is the Knowledge Graph Change Language (KGCL),
  a LinkML model that describes edit operations on a knowledge graph or
  ontology. It does so by having more than 50 classes that each
  represent a specific type of editing operation (renaming a node,
  replacing a predicate, adding a synonym, etc.); all those classes are
  grouped under a generic \emph{Change} class.

  A typical KGCL file would contain one or several instances of any of
  those classes, but crucially, we don't know in advance the exact type
  of each of those instances --~we only know that they will be instances
  of some subclasses of \emph{Change}. The exact subclass has to be
  discovered at runtime by looking at the value of the \emph{type} slot,
  which has been defined in the KGCL model as the \emph{type designator}
  for the \emph{Change} class. For example, we know that the first
  change here is an instance of \emph{NodeRename}, because that's what
  the \emph{type} slot tells us.

  But a generic YAML deserialisation library does not know what a type
  designator slot is. If you ask it to deserialise that file as a list
  of \emph{Change} objects, it will predictably expect to find instances
  of \emph{Change} --~and so, for example, it will not know what to do
  with the \emph{old\_value} and \emph{new\_value} slots, which do not
  exists in the top-level \emph{Change} class.}
\end{frame}

\begin{frame}[fragile]
  \frametitle{Limitations of generated code: the case of LinkML}

  \begin{columns}[t]
    \begin{column}{.45\textwidth}
    LinkML model of \emph{ClassDefinition}:
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
classes:
  ClassDefinition:
    slots:
      - slots

slots:
  slots:
    range: SlotDefinition
    multivalued: true
\end{Verbatim}
    \end{column}
    \begin{column}{.45\textwidth}
    Corresponding Java code:
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
public class ClassDefinition {
    List<SlotDefinition> slots;
}
\end{Verbatim}
    \end{column}
  \end{columns}

  \bigskip

  The \emph{slots} slot expects a list of \emph{SlotDefinition} objects
  but is actually serialised as a list of \emph{slot names}. Correctly
  parsing that requires knowing that:

  \begin{itemize}
    \item \emph{SlotDefinition} represents \emph{identifiable objects}
    (because it has an \emph{identifier slot});
    \item identifiable objects are serialised as \emph{references} by
    default (unless the slot holding them is explicitly \emph{inlined},
    which is not the case of \emph{slots}).
  \end{itemize}

  \medskip
  Both are LinkML concepts that a generic library like Jackson cannot
  know about.

  \note{The second example is LinkML's own meta-schema. A class
  definition in LinkML contains, among other things, a slot called
  \emph{slots}, which is expected to contain the list of slot
  definitions used by the class. This is rendered in Java,
  unsurprisingly, as a \emph{ClassDefinition} class with a field typed
  as a list of \emph{SlotDefinition}.

  But in the YAML representation of a LinkML schema, the content of the
  \emph{slots} slot is serialised, not as a list of SlotDefinition
  objects, but as a list of \emph{names}, each name being a reference to
  the actual SlotDefinition which is stored elsewhere in the YAML file.

  This is completely normal, because (1) a SlotDefinition is a globally
  identifiable object (because it has an identifier slot), and (2) the
  \emph{slots} slot is not defined as being \emph{inlined}, so its
  values are serialised by references instead. This is standard LinkML
  behaviour.

  But again, a non-LinkML-aware library would know nothing about that,
  and would error out when trying to deserialise a ClassDefinition
  because a list of names (a list of strings) is not the same thing as a
  list of SlotDefinition objects.}
\end{frame}

\begin{frame}
  \frametitle{Limitations of generated code: conclusion}

  \begin{block}{}
    Developers in language \emph{X} need a LinkML runtime library for
    the same reason than Python developers need
    \texttt{linkml\_runtime.loaders} and the like: because some LinkML
    features require more than a mere translation of a LinkML schema
    into a set of classes.
  \end{block}

  \bigskip
  To be able to read/write LinkML-defined data, a LinkML runtime library
  for any language must offer support for, at the very least:

  \begin{itemize}
    \item type designator slots;
    \item globally or locally identifiable objects;
    \item representing objects by reference;
    \item dealing with the various forms of inlining;
    \item dealing with prefixes and CURIEs.
  \end{itemize}

  \note{Overall, working with LinkML-defined instance data in Java, or I
  would argue in any other language, requires support from a runtime
  library for the same reason than Python developers have a
  linkml-runtime package: because merely translating a LinkML schema
  into a set of classes (or whatever can be used to represent classes in
  your target language) is not enough to support all LinkML features.

  In particular, developers need explicit support for at least the
  following features, which I believe represent the minimal feature set
  to work with LinkML data:

  -- the notion of type designator slots;

  -- the notion of globally or locally identifiable objects (objects with
  an identifier or key slot);

  -- the fact that globally identifiable objects can be serialised as mere
  references, rather than inlined;

  -- the various forms of list or dict inlining;

  -- and I would add to that, support for not having to explicitly deal
  with CURIES.}

\end{frame}

\begin{frame}[fragile]
  \frametitle{Alternative to a runtime library?}

  Generated code is not enough? Then let us generate \emph{more} code!

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
public class Point {
    String name;
    int x;
    int y;

    public static Point fromJSON(File f) {
        // All code required to load a Point from a JSON file
    }a

    public void toJSON(File f) {
        // All code required to dump a Point to a JSON file
    }
}
\end{Verbatim}

  \begin{block}{Not sustainable}
    \begin{itemize}
      \item Java code embedded within a Jinja2 template within a Python project
      \item A bug in the loading or dumping code? Re-generate everything
      \item Better to keep the generated code minimal and deport the
      logic elsewhere
    \end{itemize}
  \end{block}

  \note{So the code currently generated by LinkML's Java code generator
  is not enough. Couldn't we get away with that solely by making the
  generator produce more code, rather than having to use a runtime
  library?

  Yes, we \emph{could} amend the generator so that, instead of producing
  simple classes that are nothing more than data holders, it produces
  classes with additional methods providing all the logic needed to,
  minimally, load and dump instance data. Those methods would contain
  all the code needed to deal with the minimal set of features I
  mentioned on the previous slide.

  That would be possible, but I'd argue that it would not be
  \emph{practical} and in effect would be unsustainable. If only because
  we would have to develop, maintain, and debug Java code that is
  embedded within Jinja2 templates which are themselves in the middle of
  a pure Python project. As someone who is already maintaining an
  unrelated project that involves a lot of code embedded within Jinja2
  templates, I can tell you that this is a nightmarish perspective.

  I believe it is much better to keep the generated code as simple as
  possible, and deport most if not all of the logic needed to work with
  it elsewhere --~in a runtime library.}

\end{frame}

\section{Design decisions for the LinkML-Java runtime}

\begin{frame}
  \frametitle{No availability of the schema at runtime}

  \begin{block}{}
    The runtime must work solely with the Java code generated from the
    schema, without requiring the schema itself.
  \end{block}

  \bigskip
  {\scriptsize
  (Mostly to avoid a catch-22: If knowledge of a schema $S$ is needed
  at runtime to read instance data $S_d$

  \begin{itemize}
    \item then before we can read $S_d$ we must first read $S$;
    \item since $S$ is an instance of LinkML's own meta-schema $L$,
      then before we can read $S$ we must first read $L$;
    \item since $L$ is an instance of $L$, then before we can read $L$
      we must first read $L$.
  \end{itemize}

  \rightarrow\hspace{.5em} We must be able to read instance data of a
  schema without having to read the schema first. $\qed$)}

  \bigskip
  This does not preclude the possibility of future additional features
  that could be enabled by knowledge of the schema at runtime. But basic
  loading and dumping of LinkML-defined instance data \emph{must not}
  require the schema.

  \note{The first, and arguably the most important, decision is that the
  runtime must not require the schema itself in order to be able to read
  or write instance data. It must be able to work with nothing more than
  the code that was generated from the schema, without the schema
  itself.

  This makes bootstrapping easier because it dispenses us from having to
  develop a full-blown custom LinkML parser when all we want to do at
  first is to parse instance of a schema --~we're not interested in the
  schema itself, at least not at this stage.

  In the future, the library will probably offer additional features
  that may require runtime knowledge of the schema. But the basic
  features of loading and dumping LinkML-defined instances data must be
  doable without such knowledge.}
\end{frame}

\begin{frame}[fragile]
  \frametitle{Reliance on Java annotations}

  \begin{block}{}
    The generated code must provide some critical pieces of information
    from the schema, in the form of Java annotations.
  \end{block}

  \bigskip
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
@LinkURI("https://example.org/myschema/MyClass")
public class MyClass {
    @Identifier
    private String id;

    @TypeDesignator
    private String type;

    @Inlined(asList = true)
    private List<AnotherClass> foos;
}
\end{Verbatim}

  \bigskip
  Work is in progress to update LinkML(-Py)'s own code generator to
  automatically produce code that is suitable for consumption by the
  LinkML-Java runtime.

  \note<1>{The second decision is a direct consequence of the first: since
  we do not have the schema at runtime, we must rely on the generated
  code to provide us with all the informations we need to be able to
  read or write instance data.

  The generated code already tells us what are the slots and their
  ranges within a class, but we also need to know, for example:

  -- which slot is the identifier or key slot, if any;

  -- which slot is the type designator slot, if any;

  -- whether a slot is expected to be inlined, and if so in which form;

  -- the URI associated to a class or a slot;

  and a few other things. This is done through a set of dedicated Java
  annotations, to be added to the generated code. Part of my work of the
  previous weeks has been to amend LinkML's Java code generator so that
  it could automatically inject those annotations.}
\end{frame}

\begin{frame}
  \frametitle{No ties to a specific YAML/JSON library}

  \begin{block}{}
    The runtime must ideally be usable with any YAML/JSON parser/writer
    without requiring a specific one.
  \end{block}

  \bigskip
  This is to facilitate integration into existing projects that may
  already depend on a given YAML/JSON library for other purposes.

  \begin{itemize}
    \item The runtime mid-level interface works solely with
    \texttt{Map<String,Object>} objects --~it doesn't matter which
    parser library (Jackson or anything else) produced the object.
    \item For convenience, a high-level interface is provided that does
    specifically use the Jackson library. But its use is entirely
    optional.
  \end{itemize}

  \note{A somewhat less important decision: the runtime must not depend
  on a specific JSON/YAML library.

  The runtime is expected to be used by several existing projects in the
  bioimaging community --~projects that may already have a JSON/YAML
  parser among their dependencies, and I don't want to force them to use
  another parser just to work with LinkML data.

  For that, the runtime does not work with JSON or YAML files directly.
  The typical workflow for data loading, for example, is that you would
  first parse the JSON or YAML file into a generic Java Map structure
  (using any parser you want), and then you pass the map to the runtime
  --~at which point it does not matter which parser produced it.}
\end{frame}

\begin{frame}[fragile]
  \frametitle{Allow for model-specific customisations}

  \begin{block}{}
    The default behaviour of the runtime must be customisable as needed
    to meet additional, non-LinkML-defined requirements.
  \end{block}

  \bigskip
  Example with KGCL:
  \begin{columns}
    \begin{column}{.35\textwidth}
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
- id: CHANGE:001
  type: PredicateChange
  old_value: BFO:0000050
  old_value_type: curie
\end{Verbatim}
    \end{column}
    \begin{column}{.60\textwidth}\small
    The \texttt{old\_value} slot is defined as string-typed, but may
    occasionally contain a CURIE instead, as indicated by the value of
    the \texttt{old\_value\_type} slot.
    \end{column}
  \end{columns}
  
  \bigskip
  This is a KGCL-specific behaviour that is not encoded in KGCL's
  LinkML-based definition, so it cannot be implemented in a generic
  LinkML runtime library.

  \bigskip
  Client code can request the use of a custom converter object to parse
  \emph{SimpleChange} objects:
\begin{Verbatim}[frame=single,fontsize=\scriptsize]
@Converter(MyCustomSimpleChangeConverter.class)
public class SimpleChange { ... }
\end{Verbatim}

  \note{Next, the runtime must be flexible enough to accomodate the
  specific needs of some models --~including needs that are not
  themselves defined in their schemas but are expressed elsewhere.

  KGCL again provides us with an example. The \emph{old\_value} slot in
  the SimpleChange class is nominally defined as string-typed, but in
  fact it has a variable type --~sometimes a string, sometimes a CURIE;
  the exact type being indicated by the \emph{old\_value\_type} slot.

  The loading code in the runtime would not know about that, since this
  is a KGCL-specific behaviour that is not encoded in the model. So we
  must be able to request the use of a custom converter object,
  specifically intended to process instances of \emph{SimpleChange},
  where we can implement the desired behaviour.}
\end{frame}

\begin{frame}
  \frametitle{Automatic expansion/contraction of CURIEs}

  \begin{block}{}
    The runtime must take care of dealing with prefixes and CURIEs so
    that the client code does not have to.
  \end{block}

  \bigskip
  Because life is too short to spend it dealing with CURIEs any more
  than what is strictly necessary:
  \begin{itemize}
    \item CURIEs must be automatically expanded upon parsing;
    \item they must be automatically contracted upon serialising;
    \item in-between, client code must be able to forget that they even
    exist.
  \end{itemize}

  \bigskip
  Previous experience with both KGCL and SSSOM convinced me that this is
  the only sane way of working with CURIEs.

  \note{Lastly, the runtime must allow the developer to completely
  forget that CURIEs even exist, by taking care of automatically
  expanding them into IRIs upon parsing and automatically contracting
  them back into CURIEs upon serialising.

  This is arguably less important in the big scheme of things, but I
  personally consider that a critical, must-have feature. As a user of
  the runtime, I do not want to have to deal with CURIEs --~that should
  be the runtime's job.

  CURIEs are a serialisation detail and should never appear in the core
  logic of an application.}
\end{frame}

\section{What is supported for now?}

\begin{frame}[fragile]
  \frametitle{Loading from JSON or YAML files}

  High-level interface\\
  (basically the pendant of LinkML(-Py)'s \texttt{linkml\_runtime.loaders}):

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
ObjectLoader loader = new ObjectLoader();
Point p = loader.loadObject(new File("point-data.yaml"), Point.class);
List<Change> kgclChangeset = loader.loadObjects(new File("kgcl-data.yaml"), Change.class);
\end{Verbatim}

  \bigskip
  Mid-level interface:

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
ConverterContext ctx = new ConverterContext();
Map<String,Object> raw = ...; // generic representation of the data
                              // as obtained from a YAML/JSON parser
Point p = ctx.getConverter(Point.class).convert(raw, ctx);
ctx.finalizeAssignments();
\end{Verbatim}

  \note{So what you can do, currently, with the initial version of the
  LinkML-Java runtime?

  Mainly, you can load and dump LinkML instance data from and to JSON or
  YAML files.

  For loading, you have a high-level interface that abstracts away all
  the LinkML details, and that can be used roughly in a similar way to
  the loaders object in the LinkML Python runtime, in that you just give
  the loader (1) the file you want to read and (2) the type of object
  you want to deserialise, and it will do everything for you.

  What I call the ``mid-level interface'' can be used if (1) you want to
  work with data that have already been read from files and put in
  memory, and/or (2) if you need finer control about what the runtime
  will do --~this will obviously require more familiarity with the
  inner workings of the runtime, which will be explained in its
  documentation but that I will not detail here.}
\end{frame}

\begin{frame}[fragile]
  \frametitle{Dumping to JSON or YAML files}

  High-level interface:

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
ObjectLoader loader = new ObjectLoader(); // Yes, ObjectLoader is badly named;
                                          // it does both loading and dumping
Point p = ...;
loader.dumpObject(new File("point-data.yaml"), p);
List<Change> kgclChangeset = ...;
loader.dumpObject(new File("kgcl-data.yaml", kgclChangeset);
\end{Verbatim}

  \bigskip
  Mid-level interface:

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
ConverterContext ctx = new ConverterContext();
Point p = ...;
Object raw = ctx.getConverter(Point.class).serialise(p, ctx);
// Then give the "raw" object to any JSON/YAML writer of your liking
\end{Verbatim}
\end{frame}

\begin{frame}
  \frametitle{Type designators}

  \begin{itemize}
    \item Multi-valued type designators are supported.
    \item For String-typed type designators, it is assumed that all the
    generated classes reside in the same Java package
      \begin{itemize}
        \item Not an unreasonable assumption...
        \item That's already the behaviour of LinkML(-Py)'s Java code generator
      \end{itemize}
    \item For URI-typed type designators:
      \begin{itemize}
        \item No limitation/assumption about the Java package\\
        (as long as the class is reachable by the current class loader)
        \item But class URIs cannot be discovered on the fly, each class
        whose URI may be used in a type designator must be explicitly
        registered first
      \end{itemize}
  \end{itemize}

  \note{A few things to be aware of.

  First, about type designators. LinkML-Java currently supports both
  single- and multi-valued type designators. However there are some
  limitations.

  If the type designator is a string, it will only work if the
  designated class is in the same Java package as the base
  class--~which should not be too much of a limitation as the current
  behaviour of the Java code generator is to put all the code in a
  single package anyway.

  If the type designator is a URI, then the designated class can live
  anywhere, but the runtime will only recognise it if it has been
  explicitly registered first.

  Those limitations may be lifted in future versions, though I do not
  consider that a priority for the time being.}

\end{frame}

\begin{frame}[fragile]
  \frametitle{Referencing / inlining}

  \begin{itemize}
    \item References to global objects are automatically resolved into
    Java references

    For example, the following:

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
my_class_definition.getSlots().get(0)
\end{Verbatim}

    does not give you merely the \emph{name} of the first slot; it gives
    you the actual object representing the slot definition, as a (Java)
    reference ($\sim$ ``pointer'') to wherever that slot is defined.

    \item Only ``non-repair list normalisations'' are implemented
  \end{itemize}

  \note{About referencing, the most important thing to be aware of is
  that the runtime will automatically resolve LinkML references --~when
  a slot contain a name pointing to another object~-- into Java
  references.

  For example, if you are working with LinkML's own meta-schema and you
  have loaded a ClassDefinition objects: what you will find in the
  \emph{slots} slot (which you can access through the \emph{getSlots()}
  accessor) will not be the names of the slot, as it would be in Python.
  Instead it will give you directly a Java reference (what in C++ we
  would call a ``pointer'') to the actual SlotDefinition object.

  This is because (1) it is more convenient (I believe resolving those
  references is completely within the remit of a LinkML runtime) and (2)
  because Java's strict typing rules actually do not give us a choice
  here. It would not be possible to store slot names (that is, strings)
  into a field intended to store SlotDefinition objects.

  About inlining, the runtime only supports what the LinkML spec calls
  ``non-repair'' normalisations. Mostly, this means that if a slot is
  expected to be serialised as a inlined list, it will be an error if it
  is actually serialised as an inlined dict, and the other way round.}
\end{frame}

\begin{frame}[fragile]
  \frametitle{Support for \texttt{extra\_slots}}

  \begin{itemize}
    \item Requires an additional \texttt{Map<String,Object>} field
    explicitly intended to store additional data:

\begin{Verbatim}[frame=single,fontsize=\scriptsize]
public class Foo {
    ...
    @ExtensionHolder
    private Map<String, Object> extraSlots;
}
\end{Verbatim}

    \item This is because you cannot dynamically add an arbitrary field
    to a class in Java (as you can do in Python).
  \end{itemize}

  \note{The runtime does support the presence of ``extra slots'' --~that
  is, JSON/YAML keys that do not correspond to a declared slot or
  attribute for the class that is being deserialised~--, but this
  requires that, during code generation, an additional field be added
  specifically to store the values of such slots.

  If that extra field is present and labelled with an
  \emph{ExtensionHolder} annotation, then any extra slot will be stored
  in that field --~and upon serialisation they will be written back as
  additional keys as they were originally found.

  Otherwise, the loader will simply silently ignore any extra slot it
  may find.}
\end{frame}

\begin{frame}
  \frametitle{``Dynamic'' enums}

  \begin{itemize}
    \item Completely unsupported for now!
    \item Treated as simple strings with no support for checking whether
    the string represents a permissible value
  \end{itemize}

  \note{Lastly, a quick note about LinkML's dynamic enumerations. Simply
  put, they are completely unsupported for now.

  This is because, as of now I am still unsure about the best way to do
  that. One thing that is clear is that they will not be represented as
  actual Java enums --~the values of a Java enumeration must necessarily
  be known at compile time, which is by design incompatible with the
  very idea of dynamic enums.}
\end{frame}


\section{What's next?}

\begin{frame}
  \frametitle{Wishlist}

  \begin{itemize}
    \item Support for import/export from/to RDF
    \item Support for dynamic enums
    \item Allow working with a schema that is only discovered at runtime
      \begin{itemize}
        \item With no pre-generated code, just the schema itself
        \item So that we can at least \emph{validate} data against a
        model that is not known in advance
      \end{itemize}
    \item Generate Java code directly from the LinkML-Java runtime
      \begin{itemize}
        \item Instead of having to rely on \texttt{linkml generate java}
        \item This will allow for easier integration of the code
        generation step into the build process of a Java project
        \item Will most likely yield a Maven and/or Gradle plugin
      \end{itemize}
  \end{itemize}

  \note{What to expect in future versions?

  There are still a lot to implement, and the roadmap for future
  versions is not fixed yet, but RDF support --~that is, the ability to
  import or export data from or to RDF, instead of JSON or YAML files~--
  will definitely be there at some point, as well as at least some
  support for dynamic enums.

  Longer term, the runtime will also allow to work with a completely
  unknown schema that is only discovered at runtime, without requiring
  that code had been pre-generated specifically for that schema.

  Once the runtime can do that, then it will most likely at some point
  include its own code generator, which will make it much easier to
  integrate the code generation step into the build process of a typical
  Java project --~this will probably take the form of a Maven plugin.
  The idea being that you can just add this plugin to your Maven
  configuration, point it to the LinkML schema you want to work with in
  your project, and the plugin will automatically generate the code in
  the early phases of the Maven lifecycle.}
\end{frame}
